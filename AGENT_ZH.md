# 文档风格指南

这是个人知识库，不是面向外人的指南文档。编写或修改文档时请遵循以下规范。

## 核心原则

- **简洁优于冗长**：聚焦核心内容，去除冗余
- **面向技术人员**：假设读者具备基础技术知识
- **原理与步骤**：优先说明原理和操作步骤，而非教程式引导

## 写作风格

### 避免

- 教程式用语："本指南将帮助你..."、"在本教程中..."
- 第二人称："你的用户名"、"你的域名"、"你可以..."
- 过于详细的前置条件（如"需要拥有 GitHub 账户"）
- 同一操作的多种方式（选择一种即可）
- 简单的示例文件（如基础 HTML 模板）
- 显而易见的概念解释

### 推荐

- 直接陈述："将 `<username>` 替换为实际用户名"
- 使用表格整理结构化信息
- 代码块尽量少注释
- 使用列表快速呈现要点
- 附上官方文档参考链接

## 文档结构

### 域名绑定类文档

1. **原理**：简要说明工作机制
2. **DNS 记录**：表格形式，包含类型、名称、值
3. **步骤**：编号列表，仅包含必要操作
4. **验证**：检查状态的命令
5. **常见问题**：表格形式，包含问题、原因、解决方案
6. **参考**：官方文档链接

### 快速开始类文档

1. **关键规则**：表格或列表
2. **步骤**：最简编号列表
3. **优先级/顺序**：如适用
4. **状态检查**：如何验证

## 格式规范

### 使用表格呈现对比信息

```markdown
| 类型 | 名称 | 值 |
|------|------|-----|
| A | @ | 185.199.108.153 |
```

### DNS 记录格式

```
类型: A, 名称: @, 值: 185.199.108.153
```

### 命令格式

```bash
dig example.com +short
```

## 语气

- 中性、客观
- 无问候语或结束语
- 无激励性语言
- 技术准确性优于友好度
